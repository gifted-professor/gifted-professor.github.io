面试题1 - 从输入一个URL地址到浏览器完成渲染的整个过程

从输入一个 URL 地址到浏览器完成渲染的整个过程

浏览器输入地址并回车 ==> 浏览器查看当前**url是否存在缓存** 并比较缓存是否过期 ==> DNS解析当前 URL 对应的IP ==> 根据**IP建立 TCP 连接（三次握手） ==> 发送http**请求 ==> 服务器处理请求 浏览器接收 HTTP 响应 ==> 浏览器解析并渲染页面 ==> 关闭 TCP 连接（四次握手）

然后其实关于这个过程 还是有很多不懂的东西

1.DNS解析

DNS解析其实就是你寻找资源的过程 因为互联网中的每一台机器都有唯一标识的IP地址 但是这是一串数字 所以就需要一个网址和IP地址的转换 也就是DNS解析

比如输入网址的时候 会先在本地的域名服务器中查找 没找到的话就去根域名服务器查找 没有的话就去com顶级域名服务器查找 以此类推 直到找到了IP地址 然后就会缓存 以供下次使用 如 www.google.com 查找过程就是

. ==> google.con. ==> www.google.com. 这个存在最后的 . 就是根域名服务器 只是一般会省略

2.发起TCP连接

TCP提供一种可靠的传输 这个过程涉及到三次握手 四次挥手

⭐TCP 协议和UDP协议

这两个协议都工作在 传输层 他们的工作就是在程序之间传输数据

这两个协议其实就是一些二进制数

🍔 区别

他们两个最大的区别就是一个是连接 一个是基于非链接 TCP 相对来讲更牛逼一点 可能这也是我之前不知道UDP但是直到 TCP 的原因吧

🍔 三次握手

第一次握手 ==> 客户端发送syn 包（ack = x）给服务器 并进入SYN_SEND 状态 等待服务器确认

第二次握手 ==> 服务器确认收到了客户端的SYN（ack = x + 1） 并且自己发送了一个SYN 包（Seq=y） 即 SYN+ACK包 此时服务器会进入 SYN_RECV状态

第三次握手 ==> 客户端收到了服务器传来的 SYN + ACK 包 向服务器发送确认包**ACK（ack = y + 1）**此包发送完毕 服务器和客户端进入 ESTABLISHEND 状态 完成三次握手

握手过程中传送的包不包含数据 三次握手完毕之后 客户端和服务器才开始正式的传递数据 理想状态该下 TCP连接一段建立 在通信双方中任何一方主动关闭连接之前 TCP 链接都会呗一直保持下去
❓ 为什么不采用两次或者四次握手

通俗的话来讲 如果两次握手的话 syn 包在传输的过成功可能会出现传送延缓的状态 这个时候服务器可能会因为没有收到 syn 包重新发送一个 第二个syn 包服务器收到之后会发送一个ack包 表示确认 但是这个时候如果之前延缓的syn 包重新发送出去了 服务器就会以为客户端有发送了一个新的请求 想客户端发送确认 但是此时客户端不回理会 服务器就会一直等待 导致服务器的资源浪费

**四次握手的话就是因为实在是没必要 **

🍔 四次挥手

如果客户端主动发送关闭连接请求 需要先发送一个 FIN 包 标识要关闭连接 自己进入 终止等待 1 状态

服务端收到了 FIN 包 发送一个ACK包 标识自己进入关闭等待状态 客户端进入终止等待 2 状态

服务端此时可以继续发未发送的数据 客户端还可以继续接收数据 等服务端发送完数据之后会发送一个 FIN 包进入最后确认状态

客户端确认之后 发送一个 ACK 包 进入超时等待状态并关闭连接 服务端收到 ACK 包之后立即关闭连接

为什么ACK包需要等待超级等待链接

这是为了保证对方已经收到ACK包 如果客户端发送完ACK包之后就释放了链接 服务端就会一直处于等待确认状态 但是有了超时等待时间的话 服务端在没有收到 ACK包的时候会重新发送一个FIN包并刷新超时时间

UDP和TCP的区别

udp 适合传送大量的数据 不是很在乎是否会丢包 实时性高

tcp 适合传送需要精准送到的数据 实时性不是很搞
